# 上下文长度超出处理范围的应对策略

## 问题说明

大语言模型（LLM）在处理对话时存在上下文窗口限制，通常从几千到几十万 token 不等。当对话内容过长超过模型的处理能力时，会出现：
- 模型遗忘早期对话内容
- 响应不完整或截断
- 输出质量下降

---

## 应对策略

### 1. 会话管理策略

#### 1.1 定期总结与重置
```
定期将会话中的关键信息提取为结构化笔记，
然后开启新会话，将关键上下文作为系统提示或首条用户消息。
```

**操作步骤：**
1. 在对话达到上下文限制的 80% 时，触发总结
2. 提取关键决策、已完成任务、待办事项
3. 将总结作为新会话的初始输入

#### 1.2 上下文压缩
```
将冗长的对话历史压缩为精简版本，保留核心要点。
```

**示例：**
```
原始对话（100条消息）→ 压缩为 → 关键决策清单 + 当前进度
```

---

### 2. 任务拆分策略

#### 2.1 纵向拆分
```
将大型任务按步骤拆分，每个子任务在独立会话中完成。
```

**适用场景：** 大型重构、批量代码修改、文档编写

**示例（代码重构）：**
- 会话1：分析现有代码结构，输出重构计划
- 会话2：执行核心模块重构
- 会话3：执行辅助模块重构
- 会话4：验证和测试

#### 2.2 横向拆分
```
按功能模块将任务拆分，不同模块并行处理。
```

**适用场景：** 多模块项目、多个独立功能开发

---

### 3. 外部存储策略

#### 3.1 知识库管理
```
使用外部文件存储持久化信息：
- 项目笔记 (.md)
- 任务清单 (.md / .txt)
- 代码上下文说明
```

**文档命名规范：**
```
docs/
├── 项目概述.md          # 项目背景、目标、技术栈
├── 开发进度.md          # 当前进度、已完成/待办
├── API文档.md           # 接口定义、数据结构
├── 架构设计.md          # 系统架构、模块关系
└── 会话总结_YYYYMMDD.md # 每日/阶段会话总结
```

#### 3.2 文件引用模式
```
在每个新会话开始时，读取相关文档作为上下文：
1. 读取 docs/项目概述.md
2. 读取 docs/开发进度.md
3. 读取 docs/会话总结_*.md
```

---

### 4. 提示工程优化

#### 4.1 精简系统提示
```
避免冗长的系统提示，使用简洁的指令集。
```

**示例：**
```markdown
# 简洁版（推荐）
角色：Java后端开发专家
工作目录：/path/to/project
当前任务：[从进度文档读取]

# 冗长版（避免）
你是一位经验丰富的Java高级工程师，精通Spring Boot、MyBatis、Redis...
```

#### 4.2 增量上下文加载
```
只加载与当前任务相关的文件，避免一次性加载全部代码。
```

**原则：**
- 按需读取文件
- 使用 `grep` / `search_content` 定位关键代码
- 避免 `cat` 整个文件

## 实施流程

### 新会话启动检查清单

```markdown
1. [ ] 读取 docs/项目概述.md（若存在）
2. [ ] 读取 docs/开发进度.md（若存在）
3. [ ] 读取最近的会话总结（docs/会话总结_*.md）
4. [ ] 说明当前任务目标
5. [ ] 开始工作
```

### 每日收尾检查清单

```markdown
1. [ ] 更新 docs/开发进度.md
2. [ ] 记录已完成的关键决策
3. [ ] 记录未解决的问题和待验证点
4. [ ] 创建会话总结（如有必要）
```

## 总结

当遇到上下文长度限制时，核心思路是：
1. **主动拆分** — 不等达到限制，主动拆分任务
2. **外部存储** — 将关键信息持久化到文档
3. **精简加载** — 只加载当前必需的内容
4. **流程规范** — 建立会话启动和收尾的标准化流程
